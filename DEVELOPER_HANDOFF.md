- Disposal Bulk Approval is an approver-only queue efficiency layer at `/disposal`, not a second lifecycle: users need `disposal:approve`, SOD still blocks requester/creator self-approval, and the API independently checks authentication, permission, active/pending state, asset lifecycle, and policy. A selection contains only currently loaded eligible queue rows, never spans pages, and is limited to 50. The server preview groups eligible and blocked rows without mutation; it is advisory, and commit re-reads/revalidates every request. Each successful request commits independently with its own approver/timestamp, movement, audit record, and affected parent-batch status, so a concurrency change or processing failure produces an item-level blocked/failed partial result without rolling back other approvals or duplicating records. One optional shared approval remark is copied only to successful requests and does not overwrite their sale/salvage values, evidence, type, or other request-specific data. Bulk reject remains intentionally unavailable. Desktop uses queue-table selection and a contextual toolbar; mobile uses an explicit selection mode in the list flow, not an extra fixed bottom bar, so it does not collide with Mobile Field Navigation.
- Disposal Bulk Execution is a separate `disposal:edit` efficiency layer available only in the approved/awaiting-execution queue. Operators explicitly enter selection mode and may choose up to 20 currently loaded requests of one disposal type. Preview and commit use the same locked shared execution date, executor, final Disposed/Retired status, optional shared recipient, and optional historical-exception values. For `sell`, `donate`, and `dispose`, the shared recipient is a fallback only: it supplies the effective execution recipient when the authoritative request recipient is blank and never overwrites a nonblank request value. Every Preview/Commit item includes `recipientName` and `recipientSource` (`request`, `shared`, or `null`) so the effective value is reviewable before Commit. Each request otherwise keeps its own document, sale/salvage values, execution detail, evidence, movement, and audit history. Commit revalidates each request in an independent serializable transaction, preserves preview-blocked rows in the final result, supports partial success, and retries only stable deduplicated unresolved IDs. The historical no-evidence exception is shown only to `system_admin` when every selected request has zero active item and inherited batch evidence, with the existing 20-2,000 character reason and acknowledgement. No schema or migration is added by bulk execution.
